# hsadmin-ng Agent Instructions

Use these instructions when working in this repository.

## Project Context

### Business Background

- This is `hsadmin-ng`, a Java Spring Boot application for the administration of Hostsharing's:
  - User-Accounts, User-Groups, and a ReBAC system (for legacy reasons often still refered as 'RBAC') 
  - Business-Partners including person and contact data
  - Cooperatitve-memberships, -shares, and assets 
  - Hosting-Bookings and Hosting-Assets

## Technical Background
 
- The environment's Java version is just used to run Gradle.
- The Java version used for the build is determined by the Gradle build configuration.
- The build uses Gradle Kotlin DSL via `./gradlew`; the main build file is `build.gradle.kts`.
- Main Java package: `net.hostsharing.hsadminng`.
- Important modules:
  - `rbac`: actually a ReBAC-system comprising RBAC context, subjects, roles, grants, and RBAC SQL generation.
  - `hs/accounts`: Hostsharing account API and account persistence.
  - `hs/office`: office-domain entities such as persons, relations, partners, memberships, bank accounts, and mandates.
  - `hs/booking`: booking projects, booking items, and validation
  - `hs/hosting`: hosting assets and validation.
- Persistence is PostgreSQL with Liquibase SQL changesets under `src/main/resources/db/changelog`.
- The JUnit-based integration tests use PostresQL via Testcontains (Docker).
- REST APIs are described in OpenAPI YAML under `src/main/resources/api-definition`.
- Spring controller interfaces and resource DTOs are generated by `./gradlew openApiGenerate` into `build/generated/sources/openapi`.

## Architecture And Code Boundaries

- Treat OpenAPI YAML as the source of truth for REST API signatures and generated resources.
- After changing API YAML, run `./gradlew openApiGenerate` and adapt the Spring controller implementations.
- Do not manually edit generated files under `build/generated`.
- Controller implementations live in `src/main/java/...` and implement generated `*Api` interfaces.
- Keep changes close to the affected module. Avoid unrelated cleanups and broad refactorings.
- Prefer existing repository, mapper, context, and test patterns over introducing new abstractions.
- ReBAC database behavior is often implemented in SQL. Check changelogs and tests before changing Java-only behavior.
- The RBAC (ReBAC) code is generated by a Java internal DSL though, see methods named `rbac` in classes ending with `RbacEntity`
- Native queries for PostgreSQL enum types may need explicit casts such as `cast(:type as rbac.SubjectType)`.

## Liquibase Changeset Rules

- Generally, existing changesets must not be changed after they have been released. But there are exceptions, as follows.
- Add a new follow-up changeset for amendments to released database structures, for example `alter table` after an earlier changeset with the related `create table`.
- Existing changesets may be changed if they are marked with `runOnChange:true validCheckSum:ANY` and are repeatable.
- A changeset is repeatable only if it defines or updates procedures, functions, triggers, views, or similar database code using `create or replace`, `drop ... if exists`, guarded creation, or equivalent idempotent patterns.
- Do not add a separate changeset just to update existing procedures, functions, triggers, or views if the existing repeatable changeset can be updated safely.
- If a new changeset is needed for a related database change, prefer placing it in the same changelog file after the original related changeset instead of creating a new changelog file.
- Do not mix repeatable database code such as procedures or functions with tables, types, or other non-repeatable schema objects in the same changeset.
- Existing mixed changesets are a legacy. Keep them working, but do not use them as a pattern for new changesets.

You can check Liquibase migration compatibility by running the tests 
`LiquibaseCompatibilityIntegrationTest` and `ImportHostingAssets` via the tag "migrationTest"
as follows: `. .tc-environment; ./gradlew migrationTest`.

## Coding Conventions

- Follow existing Java style and formatting. Run `./gradlew spotlessApply -x test` for Java formatting.
- Use Java 21 language features.
- Use Lombok according to the existing code-style.
- Prefer Lombok `val` instead of `final var` for local inferred variables.
- Keep comments sparse and useful. Prefer expressive and readable code to explanatory comments.
- Preserve existing transaction boundaries and security annotations.
- Use `StrictMapper` and local mapping conventions instead of handwritten mappings unless the local code already does so.
- Do not introduce new dependencies without a clear need.
- Do not create new files for helper classes that are used from only one file. Put such helper classes into the file where
  they are used, similar to Kotlin-style colocated helper classes.

## Testing Guidance

- Add or update focused tests for behavior changes.
- Always source `.tc-environment` for test runs, without printing its contents.
- Prefer targeted Gradle test runs while developing:
  - `./gradlew test --tests fully.qualified.TestClass`
  - `./gradlew test --tests fully.qualified.TestClass --tests fully.qualified.OtherTestClass`
  - `./gradlew generalIntegrationTest --tests fully.qualified.TestClass` for tests tagged `generalIntegrationTest`
- Useful checks:
  - `./gradlew openApiGenerate`
  - `./gradlew test --tests ...`
  - `./gradlew spotlessJavaCheck -x test`
- Never run a full `./gradlew test`, it's extremely slow.

## Local Environment Notes

- Development aliases are documented in `README.md` and initialized from `.aliases`.
- Some Gradle targets should be run with excludes to be fast, see `.aliases` for examples. 
- `gw` is commonly used as an alias for `./gradlew` after sourcing `.aliases`.
- Testcontainers and PostgreSQL setup are documented in `README.md`.
- Do not assume local secrets or `.env` values are safe to read, print, or commit.

## Git And File Safety

- The worktree may contain unrelated local changes. Do not revert, overwrite, or reformat files outside the requested task.
- Before large edits, inspect the target files and nearby patterns.
- If large edits are really necessary, suggest the change to the user before actually applying it; let the user decide. 
- Avoid destructive commands such as `git reset --hard`, `git checkout -- <file>`, or broad deletes unless explicitly requested.
- Keep generated artifacts out of source control unless the repository already tracks them.

## Definition Of Done

- API changes are represented in OpenAPI YAML.
- Generated API signatures have been refreshed with `./gradlew openApiGenerate` when needed.
- Java controller/repository/service changes compile against the generated interfaces.
- Tests cover the changed behavior at the appropriate level.
- The PR documentation under `doc/PR` is updated when the task belongs to a tracked PR checklist.
- Formatting checks pass or any inability to run them is clearly explained.
