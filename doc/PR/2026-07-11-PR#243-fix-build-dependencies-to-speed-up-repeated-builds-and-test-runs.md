# PR#243: Fix Build-Dependencies to Speed Up Repeated Builds and Test-Runs

## Related Links (Hostsharing-internal)

- Taiga-Story: n/a (build maintenance)
- GitEA-PR: https://dev.hostsharing.net/hostsharing/hs.hsadmin.ng/pulls/243

## The Problem

Running a single pure unit test, e.g.

```shell
gw unitTest --tests net.hostsharing.hsadminng.mapper.ArrayUnitTest
```

took about 20 seconds before the test even started, while the test itself runs in about 1 second.
Even a fully "warm" run without any source changes took ~21s.

The cause was a domino effect in the build graph, triggered on every single invocation:

1. All 7 `processSpring*` OpenAPI-processor tasks wrote into the **same** target directory
   `build/generated/sources/openapi-javax`. Such overlapping outputs defeat Gradle's
   up-to-date checks, so at least one processor re-ran on every build.
2. That changed the inputs of the `openApiGenerate` copy task (the `javax`→`jakarta` rename),
   which re-created all generated `.java` files with new timestamps.
3. `compileJava` saw changed sources and recompiled the complete main code base.
4. `compileTestJava` recompiled the complete test code base.
5. Only then the actual 1-second test ran — followed by an unconditional `jacocoTestReport`,
   because `unitTest` had a `finalizedBy(jacocoTestReport)`.

## The Requirements

- Repeated test runs without source changes must not regenerate OpenAPI sources
  or recompile the main/test code base.
- The JaCoCo report is only generated when explicitly requested as a Gradle target
  (`gw jacocoTestReport`); `gw check` and `gw jacocoTestCoverageVerification` keep working.
- The Gradle build cache is enabled to also reuse task outputs across builds
  (e.g. after `rm -rf build` or branch switches).

## Background

The OpenAPI sources are generated by openapi-processor-spring 2022.5, which still emits
`javax` imports. Therefore, the extra `openApiGenerate` copy task renames `javax` to `jakarta`
while copying from `generated/sources/openapi-javax` to `generated/sources/openapi`,
and only the latter is attached to the main source set.
This two-step generation is why stale timestamps in the intermediate directory
cascaded into fully recompiling.

## The Solution

- `build.gradle.kts`: each `processSpring*` task now writes into its own target directory
  `build/generated/sources/openapi-javax/<processName>`. Without overlapping outputs,
  Gradle's up-to-date checks work again and the processors are skipped when their
  API YAML files are unchanged.
- `openApiGenerate` copies from each per-processor directory, so the process names
  do not end up in the generated package paths. The generated modules use distinct
  Java packages, hence no duplicate relative paths can collide.
- `HsadminTestTasksPlugin.kt`: removed `finalizedBy(jacocoTestReport)` from the `unitTest`
  task. The report task itself still depends on `unitTest`, so `gw jacocoTestReport`
  and the `check`/`jacocoTestCoverageVerification` wiring behave as before.
- `gradle.properties`: enabled the local Gradle build cache via `org.gradle.caching=true`.

Measured locally (warm run of a single unit test, no source changes):

| Scenario                                   | Duration | Comment                                      |
|--------------------------------------------|---------:|----------------------------------------------|
| before, `gw unitTest --tests ArrayUnitTest` |     ~21s | full OpenAPI regenerate + recompile + JaCoCo |
| after, same command                         |    ~2.7s | 23 of 25 tasks up-to-date                    |

Note: with completely unchanged inputs, `unitTest` itself is also up-to-date and the test
is not re-executed; use `--rerun` to force execution.
